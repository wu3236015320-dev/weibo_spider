# 微博爬虫与数据分析系统

一个基于 Python 的微博数据采集和分析工具，核心解决三个问题：**如何稳定地爬取数据**、**如何高效地存储数据**、**如何直观地分析数据**。

---

## 设计思路

### 1. 反爬虫策略 - 如何让程序像真人一样浏览

微博有严格的反爬虫机制，直接请求会被识别为机器人。我们的解决方案是**"三层伪装"**：

#### 第一层：浏览器层面伪装
- 使用 Selenium 启动真实的 Chrome 浏览器，而非直接发送 HTTP 请求
- 随机更换 User-Agent（`fake-useragent` 库提供真实浏览器的 UA）
- 禁用 Chrome 的自动化标记（如 `enable-automation` 开关）
- 通过 CDP（Chrome DevTools Protocol）注入脚本，抹除 `navigator.webdriver` 属性

#### 第二层：行为层面伪装
- **随机延迟**：每次操作后随机等待 1-3 秒，模拟人类阅读时间
- **平滑滚动**：滚动页面时随机选择 300-800px 的距离，分多次完成，模拟鼠标滚轮
- **换页间隔**：翻页时等待 60-180 秒（可配置），避免触发频率限制（其实可以自己改）

#### 第三层：异常处理
- 检测到登录页面时自动暂停，弹出提示等待用户扫码
- 每个 DOM 查询都包裹 try-except，单条解析失败不影响整体流程
- 失败时自动保存截图和 HTML，便于排查问题

### 2. 数据存储设计 - 为什么要用 SQLite

选用 SQLite 而非 MySQL/PostgreSQL 的原因：

| 维度 | SQLite | 其他数据库 |
|-----|--------|----------|
| 部署成本 | 零配置，单文件 | 需安装服务、配置连接 |
| 便携性 | 整个数据库就是一个文件 | 需要导出/导入 |
| 数据量 | 本项目万级数据足够 | 百万级才需要 |
| 查询性能 | 本地磁盘 IO，无网络开销 | 存在网络延迟 |

**表结构设计：**
- 单表 `weibo_posts` 存储所有微博
- 字段覆盖：发布者、内容、互动数据（赞/评/转）、发布时间、爬取时间、链接
- 主键自增 ID，按 ID 倒序排列（最新在前）

**分页加载机制：**
- GUI 启动时不加载数据，切换到对应页面才触发查询
- 每页 100 条，通过 SQL 的 `LIMIT/OFFSET` 实现服务端分页
- 避免一次性加载全部数据导致界面卡顿

### 3. GUI 架构 - 为什么选择 PyQt5

懒

#### 信号槽机制解决线程问题
爬虫操作耗时较长（需要等待页面加载、滚动、延迟），如果在主线程执行会导致界面冻结。

解决方案：
- 将爬虫封装在 `QThread` 子线程中
- 通过 `pyqtSignal` 与主线程通信：
  - `finished` 信号：爬取完成时传递数据列表
  - `error` 信号：发生异常时传递错误信息
  - 主线程收到信号后更新界面，全程不阻塞

#### 页面组织方式
使用 `QStackedWidget` 实现单窗口多页面：
- 首页：统计概览 + 功能导航
- 查询页：关键词搜索 + 分页浏览
- 新增页：手动录入数据
- 管理页：编辑/删除/批量清空
- 可视化页：图表分析

左侧固定菜单栏，点击切换右侧内容区，体验类似现代 Web 应用。

### 4. 可视化分析 - 如何从文本中提取洞察

#### 中文分词处理流程
1. 使用 `jieba` 对微博内容进行分词
2. 过滤停用词（"的"、"了"、"是"等虚词 + 微博功能词如"展开"、"全文"）
3. 过滤单字词、纯数字、包含标点的词
4. 统计词频，过滤占比超过 30% 的词（通常是搜索关键词本身）

#### 停用词配置系统
- 内置默认停用词集合（虚词 + URL 相关 + 微博平台词）
- 支持通过 GUI 添加自定义停用词
- 用户自定义词持久化到 `data/stop_words.json`
- 加载时合并默认集合和用户集合（使用 Set 的并集操作）

#### 图表设计
采用 **2:1 布局**：
- 左侧 2/3：词云图（字号代表词频，直观展示讨论热点）
- 右上 1/3：TOP20 高频词条形图（精确量化）
- 右下 1/3：内容长度分布直方图（附均值/中位数参考线）

**中文字体适配：**
- 尝试加载系统字体：微软雅黑 → 黑体 → 宋体 → 楷体
- 使用 `FontProperties` 直接指定字体文件路径，避免字体名映射失败
- 统一设置 `axes.unicode_minus` 解决负号显示为方块的问题

### 5. ChromeDriver 自动管理

**核心问题**：Chrome 和 ChromeDriver 版本必须匹配，但两者都在频繁更新。

**解决方案**：
1. 读取 Windows 注册表获取 Chrome 版本
2. Chrome 115+ 版本使用 Chrome for Testing 的 JSON API 查询对应 Driver 版本
3. Chrome 旧版本使用原版本对应规则
4. 自动下载、解压、移动到 `drivers/` 目录

**降级方案**：如果自动下载失败，提供详细的手动下载指引。

## 核心模块说明

### weibo_spider.py - 爬虫引擎

| 函数 | 作用 | 关键设计 |
|-----|------|---------|
| `setup_driver()` | 初始化浏览器 | 多重反爬配置 + 本地/自动下载 Driver |
| `crawl_by_keyword()` | 主爬取流程 | 登录检测 + 分页循环 + 数据解析 |
| `human_like_delay()` | 拟人化延迟 | 随机时间增加不可预测性 |
| `parse_number()` | 数字标准化 | 处理"1.2万" → 12000 的转换 |

### database.py - 数据访问层

| 函数 | 作用 | 关键设计 |
|-----|------|---------|
| `insert_post()` | 插入数据 | 自动记录爬取时间，立即提交事务 |
| `get_posts_paginated()` | 分页查询 | `LIMIT/OFFSET` 实现服务端分页 |
| `get_statistics()` | 聚合统计 | `SUM()` 计算总互动数，用于首页展示 |

### gui_app.py - GUI 应用

| 类/函数 | 作用 | 关键设计 |
|---------|------|---------|
| `SpiderThread` | 爬虫工作线程 | 继承 `QThread`，信号槽机制通信 |
| `ChartWidget` | 图表组件 | Matplotlib 嵌入 PyQt5，自定义 GridSpec 布局 |
| `switch_page()` | 页面切换 | 懒加载策略，按需触发数据查询 |
| `show_filter_words_dialog()` | 停用词设置 | 对话框交互，JSON 持久化 |

### config.py - 配置管理

| 函数 | 作用 | 关键设计 |
|-----|------|---------|
| `load_stop_words()` | 加载停用词 | 默认集合 + 用户自定义集合的并集 |
| `save_stop_words()` | 保存停用词 | JSON 序列化，UTF-8 编码 |

## 快速开始

```bash
# 1. 安装依赖
pip install -r requirements.txt

# 2. 自动下载 ChromeDriver
python setup_driver.py

# 3. 启动 GUI
python gui_app.py
```

**使用流程：**
1. 点击左侧"开始爬取"按钮
2. 输入关键词（如"咖啡"）、起始页、结束页
3. 如需登录，按提示使用微博 APP 扫码
4. 爬取完成后切换到"数据可视化"查看分析结果

## 技术选型理由

| 技术 | 选择理由 |
|-----|---------|
| **Selenium** | 模拟真实浏览器，绕过基于 HTTP 特征的反爬 |
| **SQLite** | 零配置、单文件、足够支撑万级数据 |
| **PyQt5** | 信号槽机制成熟，matplotlib 集成生态完善 |
| **jieba** | 中文分词领域最活跃的库 |
| **WordCloud** | 支持中文、自定义形状、颜色映射 |

---

## 注意事项

1. **爬取频率**：默认每页间隔 60-180 秒，请手动修改代码加快
2. **登录状态**：微博搜索部分结果需要登录，建议提前登录并保存 Cookie
3. **数据量**：SQLite 在十万级以下性能良好，超过建议迁移到 PostgreSQL
